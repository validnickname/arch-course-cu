Recap: what is composition?

Composition = building a system by assembling smaller components rather than one monolithic block.

  • Each component has a clear responsibility
  • Components are connected by connectors
  • Architecture ≈ components + connectors

Every connector implies shared assumptions—those assumptions are what Chapter 8 calls a dependency surface.

Recap: connectors

A connector is the mechanism that allows communication between components.

  • Defines protocol, direction, and often sync vs async
  • Same component can use different connectors to different neighbors
  • Choosing the wrong connector affects coupling, latency, and failure modes

Recap: message passing

Synchronous

Caller sends a message and waits for a reply.

Simple mental model; tight temporal coupling; blocking under load.

Asynchronous

Caller sends and continues; consumer processes later.

Better decoupling and scale; harder reasoning; event schema still couples publishers and subscribers.

What is coupling?

Coupling is how much one part of a system depends on concrete details of another (API shape, behavior, timing, deployment, or data layout).

  • Strong coupling → a small change can ripple across modules, teams, or external integrators
  • Dependencies are unavoidable; the goal is to make them visible, directional, and affordable
  • Architectural views (component, deployment, runtime) should show where coupling concentrates

Coupling is the “resistance” you feel when one team cannot ship without coordinating many others.

Why coupling matters

Coupling is a first-class driver of quality attributes you care about in practice.

  • Change velocity — high coupling slows refactors and safe rollouts
  • Testability — hard to isolate units when everything reaches into shared structures
  • Reliability — sync chains and shared failure domains amplify outages
  • Shared databases, generated DTOs, and “reuse internal models in the API” are classic coupling magnets

Reviews should ask: “If we change X, who must change in the same release train?”

Reducing coupling

  • Interfaces / ports — depend on abstractions; swap adapters for DB, queue, or HTTP (Example 2)
  • Anti-corruption layer — translate external models at the boundary; don’t leak vendor types inward
  • Versioned public API — stable façade over evolving internals
  • Feature flags — decouple code present from behavior exposed
  • Tolerant reader — ignore unknown fields; optional new data (Example 1)

Events reduce some temporal coupling but move coupling to schema contracts and consumer logic.

Cohesion (brief)

Cohesion = how closely the responsibilities inside one module belong together.

  • High cohesion + loose coupling across modules is usually easier to evolve
  • Low cohesion often shows up as “god services” that drag unrelated callers into the same release
  • When splitting services, aim for clear boundaries—not just smaller deployables

Coupling is between parts; cohesion is within a part. Both affect change safety.

Compatibility

Parts are compatible when one side can evolve without forcing an immediate, coordinated change on every consumer.

  • Backward — new server / producer, old clients still correct (primary goal for public APIs)
  • Forward — old server, new clients (limited; needs feature detection, fallbacks, or negotiation)
  • Transitional — dual endpoints, aliases, or adapters during migration windows

“Compatible” is defined relative to documented client assumptions—not only JSON Schema validity.

Syntactic vs semantic

Syntactic

Wire-level shape: field names, types, required vs optional, enums, headers.

Tools: JSON Schema, OpenAPI, protobuf definitions.

Semantic

Meaning and behavior: validation ranges, ordering guarantees, idempotency, error code semantics, side effects.

Same payload can imply different behavior after a “compatible” deploy.

Example: stricter validation can be a silent break even when the schema file barely changed.

Breaking vs non-breaking (typical)

Often non-breaking

Add optional field or query param with a safe default.

Add new resource or endpoint; widen accepted values if previously undocumented.

Often breaking

Rename/remove required fields; new required headers or body fields.

Tighten validation; change default behavior; remove endpoints.

Golden question: Would a deployed old client still succeed without code or config changes?

Semantic versioning (reminder)

  • MAJOR — changes that are incompatible for existing consumers under your stated rules
  • MINOR — backward-compatible additive functionality
  • PATCH — backward-compatible bug fixes
  • Pre-release tags (-beta) signal instability; document what “compatible” means for your API

Semver only helps if your policy matches real client parsers (strict vs tolerant).

Version coexistence

  • URL paths — /v1/tasks vs /v2/tasks; easy for partners to reason about
  • Headers / content negotiation — Accept-Version, vendor media types; needs disciplined gateways
  • Separate deployments — different clusters or hostnames; strongest isolation, highest ops cost
  • Sunset — published dates, metrics on remaining v1 traffic, escalation for stragglers

Dual stacks pay rent: routing, auth, observability, security patches on both code paths.

Dependencies & layering

Keep policy and domain from depending on low-level infrastructure details (ORM, message SDK, HTTP client quirks).

  • Dependency inversion — inner layers define ports; outer layers provide adapters
  • Improves test doubles, alternative implementations, and incremental rewrites
  • Public API models need not equal internal entity graphs—translate at the edge

Walk through example2_coupling_and_dependencies.py: tight dict vs TaskRepository.

Contracts in practice

  • Treat OpenAPI / AsyncAPI / schema repos as products with owners and changelogs
  • Consumer-driven contracts — tests prove providers still satisfy real client expectations
  • Postel’s principle — conservative in what you send, liberal in what you accept (within security bounds)
  • Document error envelopes: stable code, evolving message strings

Pair with Example 1: lenient parse + additive response fields.

API & schema evolution patterns

  • Additive JSON fields with defaults; avoid reusing field names for new meanings
  • Deprecation headers (Deprecation, Sunset) + docs + metrics
  • Parallel fields during rename (done + completed) before removing the old one
  • Feature flags to expose behavior only after clients are ready
  • For events: schema registry, compatibility modes (backward/forward), replay testing

Patterns reduce pain but do not remove the need for a written policy (see assignment Part 3).

Takeaways

  • Coupling is about dependency strength and direction—not only “do they talk?”
  • Syntactic wire compatibility can mask semantic breaks (validation, defaults, errors)
  • Semantic versioning works when teams share a precise definition of “compatible client”
  • Coexistence strategies trade isolation for operational and documentation overhead
  • Boundaries (ports, ACLs, versioned façades) turn big-bang releases into staged, measurable migrations
